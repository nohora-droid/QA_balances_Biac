2Contratos MR — metodología correcta RESUELTO

Estado: Resuelto. Contratos MR coinciden en 0.00% para todos los meses analizados.

Problema inicial

En una versión anterior del script se consultaba el endpoint /contracts/monthly para obtener los contratos MR. Este endpoint devolvía valores incorrectos (0 kWh para MR en todos los meses), posiblemente porque filtra por day_type=TO que no siempre está disponible.

Solución implementada

La fuente correcta para contratos energía despachada es el campo dispatch_desp del endpoint /balance/matrices. Este array contiene registros horarios con market_type: "R" (Regulado/MR) o "N" (No Regulado/MNR).

// Endpoint correcto para contratos MR/MNR GET /balance/matrices?year=2026&month=1&version_name=TxF // Estructura dispatch_desp { date: "2026-01-01", hour: 1, contract: "84390", market_type: "N", // "R" = MR | "N" = NR (incluye compra Y venta) value: 184.86 } // kWh // Suma MR: todos los registros market_type === "R" api_cttosMR = dispatch_desp .filter(r => r.market_type === 'R') .reduce((s, r) => s + r.value, 0);
Recomendación para Tech: Documentar que dispatch_desp.market_type="R" es la fuente canónica para contratos MR despachados en la versión TxF. El endpoint /contracts/monthly sirve para catálogo de contratos, no para energía despachada.

3Contratos MNR — separación Compra vs Venta RESUELTO

Estado: Resuelto. Contratos MNR coinciden en 0.00% para todos los meses una vez aplicado el filtro de contratos Venta NR.

Problema encontrado

El campo dispatch_desp market_type="N" incluye tanto contratos de Compra MNR (lo que BIAC compra) como contratos de Venta NR (lo que BIAC vende a sus clientes no regulados). Si se suman todos los registros market_type="N" directamente, el total excede el valor del sheet en un 9–22% dependiendo del mes.

Solución implementada

Se consulta el endpoint /contracts (catálogo) para obtener los sic_code de todos los contratos con operation="Venta" y type="NO REGULADO". Luego se excluyen esos sic_code del filtro en dispatch_desp.

// 1. Obtener sic_codes de contratos Venta NR desde catálogo GET /contracts?limit=500&offset=0 // Contratos Venta NR identificados (vigentes a 2026): // sic_code 88305 = OM202511NTCG-01 (BIAC) // sic_code 88867 = BIAC-12262025 (BIAC) // sic_code 88963 = 202512DEPI-01 (BIAC) // sic_code 88734 = 202511BTG-01 (BIAC) // sic_code 88503 = 202511RTAC-01 (BIAC) const ventaNRSicCodes = new Set( contracts .filter(c => c.operation === 'Venta' && c.type === 'NO REGULADO') .map(c => c.sic_code) ); // 2. Filtrar dispatch_desp para obtener solo Compra MNR api_cttosMNR = dispatch_desp .filter(r => r.market_type === 'N' && !ventaNRSicCodes.has(r.contract)) .reduce((s, r) => s + r.value, 0);
Atención: La lista de contratos Venta NR puede cambiar si se registran nuevos contratos de venta. El script consulta /contracts en cada ejecución, por lo que se mantiene actualizado automáticamente. Si se agrega un nuevo contrato de venta NR, el QA lo detectará correctamente sin cambios en el script.
Recomendación para Tech: Considerar agregar un campo operation_type directo en dispatch_desp (e.g. "COMPRA" / "VENTA") para que el consumidor de la API no necesite cruzar contra el catálogo de contratos. Esto simplificaría la consulta y reduciría el riesgo de errores de cruce.

4Demanda: diferencia sistemática ~1.4–1.9% PENDIENTE ACLARACIÓN

Estado: Diferencia identificada y explicada metodológicamente. No es un error de datos — es una diferencia de concepto entre fuentes. Se requiere decisión del equipo sobre cuál usar como referencia.

Hallazgo

La demanda regulada TxF del sheet excede consistentemente a la que reporta la API en un rango de 1.37% a 1.87% para todos los meses. La demanda no regulada muestra la misma diferencia sistemática.

Causa probable

Las dos fuentes miden puntos distintos del sistema de transmisión:

  • Sheet (TxF): Demanda medida en el punto de entrega al cliente, incluyendo pérdidas en la red de STN (Sistema de Transmisión Nacional).
  • API (DMRE/DMNR): Demanda medida en el punto de PTB (Punto de Transferencia de Bolsa), que es antes de las pérdidas STN.
  • La diferencia de ~1.5% representa aproximadamente las pérdidas en el STN, que es un valor típico para el sistema colombiano.
Recomendación: Definir formalmente cuál es la fuente de verdad para la demanda en los reportes de balance:
  • Si el balance debe reflejar lo entregado al cliente → usar el valor del sheet (TxF con STN)
  • Si debe reflejar lo comprado en bolsa/contratos → usar API DMRE/DMNR (PTB)
  • Documentar esta decisión en el procedimiento de cierre mensual
⚠ No comparar Demanda Plan vs API TxF: La columna "Plan" del sheet refleja proyecciones presupuestales. La API siempre reporta valores TxF (Transacciones Financieras, i.e., valores reales). La diferencia del 12–19% que aparece en el dashboard para "Demanda Regulada Plan" es esperada y no indica un error.

7Proceso QA: cómo correr las validaciones

Requisitos

  • Node.js instalado
  • Bearer token de Firebase válido (dura 1 hora — obtener desde DevTools → Red → token?key=... → idToken)
  • Archivo de referencia data-YYYY-MM-balance.json para el mes a validar

Comando

node run-qa.js <año> <mes> "<token>" # Ejemplos: node run-qa.js 2026 1 "eyJhbGci..." # Enero node run-qa.js 2026 4 "eyJhbGci..." # Abril

Flujo del script

  1. /balance/context → obtiene versión TxF
  2. /contracts → sic_codes de contratos Venta NR (para filtrar MNR)
  3. /balance/matrices → demanda (DMRE/DMNR) + contratos (dispatch_desp)
  4. /balance/reconciliation → bolsa compra/venta (official_balcttos)
  5. Compara vs data-YYYY-MM-balance.json y guarda resultado en qa-results/
